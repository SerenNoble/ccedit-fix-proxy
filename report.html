ccedit-fix-proxy
Claude Code Edit 修正代理

轻量级 SSE 正向代理，实时拦截并自动修正 GLM 等模型生成 Edit 调用时的 old_string 不匹配问题，使非原生 Claude 模型在 Claude Code 中获得可靠的文件编辑能力。

01 — Background
问题背景
Claude Code 通过 Edit Tool 实现精确文件替换——模型输出 old_string 和 new_string， Claude Code 在文件中精确查找 old_string 并替换。但 GLM 等模型生成的 old_string 经常与文件实际内容存在细微差异：
␣
多余空格
原文 name="test" → 模型输出 name = "test"
↵
多余空行
模型在函数间插入不存在的空白行
⇥
Tab / 空格混用
文件用 4 空格缩进，模型输出 Tab
¶
换行符差异
文件使用 \r\n，模型输出 \n
→
缩进偏移
模型多缩进或少缩进了若干层级
''
弯引号 / 直引号
文件使用弯引号 ""，模型输出直引号 ""

04 — Matching Engine
两级匹配引擎
7 种匹配策略。层级 1 复刻 Claude Code 原生逻辑，层级 2 处理 Claude Code 无法覆盖的 GLM 特有问题。
①
精确匹配 Native
直接子串查找 file_content.includes(search_string)
②
引号归一化 Native
弯引号 ↔ 直引号转换，返回文件原文
①
去行尾空白
每行 rstrip() 后精确匹配
③
换行符统一
\r\n → \n，兼容 CRLF 文件
④
Tab ↔ 空格
检测文件缩进风格，Tab ↔ 2/4/8 空格互转
⑤
空白归一化 NEW
去除所有空格和空行后匹配（滑动窗口），定位后返回文件原文
⑥
缩进匹配
按行 strip 匹配内容，计算缩进差值 delta，同步调整 new_string

05 — SSE Reconstruction
SSE 事件重生成
修正成功后，代理丢弃原始缓冲事件，重新生成 3 个 SSE 事件发送给 Claude Code。
事件替换过程
丢弃
content_block_start (original)
content_block_delta × N (original)
content_block_stop (original)
生成
content_block_start
content_block_delta (fixed JSON)
content_block_stop
index 保持一致
重生成事件的 index 与原始相同，保证 Claude Code 解析正确
tool_id 保持一致
id 与原始 block 相同，保证 tool_use 关联正确
name 统一为 "Edit"
兼容模型输出 "edit" (小写) 的情况
ensure_ascii=False
保留中文等 Unicode 字符，避免二次转义

08 — Deployment
部署方式与快速启动
支持同机和跨机两种部署拓扑。
方式 A
同机部署
代理和 Claude Code 在同一台机器，localhost:8080 直接通信
方式 B
跨机部署
代理在服务器 A，Claude Code 在服务器 B，自动处理 no_proxy 绕过
快速启动

1. pip install -r requirements.txt

2. cp .env.example .env 并填写 API Key 和上游地址

3. bash start_proxy.sh 启动代理

4. bash start_cc.sh 启动 Claude Code
